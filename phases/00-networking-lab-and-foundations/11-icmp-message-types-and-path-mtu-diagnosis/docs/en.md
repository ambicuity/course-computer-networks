# ICMP Message Types and Path MTU Diagnosis with ping and traceroute

> ICMP (RFC 792) is the diagnostic backbone of IP: an 8-byte header with 1-byte **type**, 1-byte **code**, a 2-byte **RFC 1071 checksum**, a 4-byte "rest of header", and variable payload. `ping` exchanges **Echo Request (type 8)** / **Echo Reply (type 0)** with id+seq, measuring RTT; `traceroute` abuses the IP **TTL** field — probes with TTL = 1, 2, 3... make each k-th router return **Time Exceeded (type 11, code 0)**, revealing the path hop by hop. **Path MTU Discovery** (RFC 1191, IPv4; RFC 8201, IPv6) sends DF-set packets; when a link is too small the router returns **Dest Unreachable (type 3, code 4)** carrying the next-hop MTU, letting the sender step its PMTU down the plateau table (68, 1280, 1492, ...). Failure modes are **black-hole PMTUD** (a firewall drops type 3 code 4) and **stale MTU = 0** from pre-RFC-1191 routers. This lesson builds a runnable ICMP parser, checksum verifier, traceroute model, and PMTUD simulator.

**Type:** Lab
**Languages:** Python, ping, traceroute, tcpdump
**Prerequisites:** IPv4 header and TTL/DF bits, ICMP as IP protocol 1, basic MTU and fragmentation

## The Problem

A service migrates into a cloud VPC and large HTTP POSTs suddenly hang until they time out, while small `curl` requests succeed. `ping` with default 56-byte payloads reports perfect connectivity at sub-millisecond RTT. The team suspects the application, then TLS, then MTU — but cannot reproduce it because every probe is tiny.

The real cause is a Path MTU bottleneck (a VPN, PPPoE at 1492, or a GRE/IPsec tunnel subtracting overhead) plus a firewall that silently drops the ICMP type 3 code 4 the bottleneck router generates. The sender keeps transmitting 1500-byte DF-set packets that vanish into a black hole, never learning to shrink. The tools that expose this — `ping -M do -s 1472`, `traceroute`, `tcpdump 'icmp'` — rely on ICMP; reading them correctly is the difference between a five-minute fix (MSS clamping) and a week of profiling.

## The Concept

### The ICMP message format, field by field

ICMP travels inside IP with **protocol number 1** (IPv4) or **Next Header 58** (ICMPv6, RFC 4443). Each ICMP message shares an 8-byte base:

| Offset | Field | Size | Meaning |
|---|---|---|---|
| 0 | Type | 1 byte | 0 Echo Reply, 3 Dest Unreachable, 8 Echo Request, 11 Time Exceeded |
| 1 | Code | 1 byte | Sub-class (type 3 has codes 0–15; code 4 = Frag Needed) |
| 2 | Checksum | 2 bytes | RFC 1071 one's-complement sum, field = 0 when computed |
| 4 | Rest of header | 4 bytes | id+seq (Echo), zeros (Time Exceeded), next-hop MTU (Frag Needed) |
| 8 | Payload | variable | Echo carries caller bytes; errors carry invoking IP header + 8 bytes |

The checksum is mandatory: `IcmpMessage.parse()` rejects messages whose verify value is non-zero. See `assets/icmp-message-types-and-path-mtu-diagnosis.svg` for the layout.

### The ICMP type/code registry you meet

| Type | Name | Codes | Generated by |
|---|---|---|---|
| 0 | Echo Reply | 0 | A host answering type 8 (`ping`) |
| 3 | Dest Unreachable | 0 net, 1 host, 3 port, **4 frag needed** | A router or destination host |
| 5 | Redirect | 0–3 | A better next-hop (ignored by modern hosts) |
| 8 | Echo Request | 0 | `ping` and other probes |
| 11 | Time Exceeded | 0 TTL expired, 1 reassembly | A router that decremented TTL to 0 (`traceroute`) |
| 12 | Parameter Problem | 0 | A malformed IP header |

Type 3 code 4 drives PMTUD; type 11 code 0 drives `traceroute`.

### ping: Echo Request and Echo Reply

`ping` sends ICMP **type 8, code 0**. The 4-byte "rest of header" is a 2-byte **identifier** (the sender's PID, for multiplexing pings) and a 2-byte **sequence number**. The receiver copies id, seq, and payload verbatim into a **type 0, code 0** Echo Reply; the RTT is the wall-clock gap (matched on id+seq). The `-s` flag sets payload size; on Linux `-M do` sets the IP **DF** bit so oversized probes return type 3 code 4 instead of being fragmented — the foundation of the manual PMTUD probe.

Worked probe: `ping -M do -s 1472 203.0.113.10`. 1472 + 8 ICMP + 20 IP = 1500 bytes. If the path supports 1500, an Echo Reply returns. If a downstream link is 1492, the router returns **type 3 code 4 with next-hop MTU 1492**. Drop `-s` to 1464 and it succeeds.

### traceroute: exploiting TTL and Time Exceeded

`traceroute` abuses the IP **Time to Live** field — no special ICMP type. It sends a probe (UDP to a high port on classic traceroute, ICMP Echo on Linux `-I`, or TCP SYN on `tcptraceroute`) with **TTL = 1**. The first router decrements TTL to 0, drops the packet, and returns **ICMP type 11 code 0** with its own source address — revealing hop 1. It retries with TTL = 2 (the second router expires it), and so on until the destination answers with the application-level reply (Echo Reply, SYN-ACK, or port unreachable).

| Probe TTL | Expires at | ICMP returned | What you learn |
|---|---|---|---|
| 1, 2, ... | k-th router | type 11 code 0 from Rk | Rk's address + RTT |
| n | destination | type 0 (or port unreachable) | path complete |

`code/main.py`'s `traceroute()` prints TTL, router address, ICMP type/code, and RTT per hop. Caveats: a router may answer TTL-expiry from a different interface than the probe arrived on (so a hop's address shifts), and some hops silently skip Time Exceeded (the `* * *` lines), often from rate limiting or a firewall.

### Path MTU Discovery: DF, plateau table, rounding

A host avoiding fragmentation runs **PMTUD** (RFC 1191, IPv4; RFC 8201, IPv6). It sends packets at its link MTU (commonly 1500) with the **DF bit set**. When a downstream link is smaller, the router drops the packet (cannot fragment because DF is set), returns **ICMP type 3, code 4**, and places the **next-hop MTU** in the "rest of header" (RFC 1191 extended RFC 792's field for the real MTU).

The sender reduces its **Path MTU (PMTU)** and retries. RFC 1191 says round the reported MTU *up* to the nearest plateau: **68** (IPv4 min, RFC 791), **296**, **508** (X.25), **1006**, **1280** (IPv6 min, RFC 8200), **1492** (PPPoE), **1500** (IEEE 802.3), **2002**, **4352/4470**, **9216** (jumbo).

Worked from `code/main.py`: probe at 1500 over PPPoE; router reports 1492; the nearest plateau ≥ 1492 is 1492, so the sender adopts 1492 and resends. If a pre-RFC-1191 router reports **0**, the sender steps *down one plateau* rather than trusting the bogus value — the `next_plateau()` fallback. IPv6 never fragments in the network, so PMTUD is mandatory with floor 1280.

### Black-hole PMTUD and the firewall problem

The most common failure is **black-hole PMTUD**: a security policy drops all ICMP, including the type 3 code 4 PMTUD depends on. Symptoms: small packets (SYN, ACK, keystrokes) pass; large packets (file upload, TLS bulk data) stall and retransmit until timeout; `ping -M do -s 1472` fails.

Diagnosis ladder:
1. `ping -M do -s 1472 <host>` — does the DF probe get an ICMP error, or just vanish?
2. `traceroute --mtu <host>` walks the path increasing payload until a hop stops — pinpoints the constraining hop.
3. `tcpdump -n icmp and host <host>` — confirms whether type 3 code 4 is generated *and received*.

Mitigations, in order of preference:
- **Fix the firewall** to permit ICMP type 3 code 4 (and type 11) — the correct fix; ICMP is not optional cargo.
- **MSS clamping** (RFC 6691): rewrite TCP MSS in SYN/SYN-ACK to PMTU − 40, so endpoints never oversize.
- **Lower the interface MTU** (e.g. `ip link set eth0 mtu 1492`) — crude but reliable.
- **Clear DF** on the sender — only viable where fragmentation in the network is acceptable (rare; IPv6 cannot do this).

### ICMP rate limiting and trust

Routers must not generate unlimited ICMP; RFC 1812 sets token-bucket rate limiting. A busy hop may return only a fraction of the Time Exceeded `traceroute` expects, producing `* * *` rows that look like loss but are the router protecting itself. ICMP error messages include the invoking IP header plus at least 8 bytes of payload, so `tcpdump` can show which TCP flow triggered a Frag Needed. ICMP Redirects (type 5) are ignored by modern hosts (a forged redirect is a trivial on-path attack) but remain in RFC 792.

## Key Terms

| Term | What people say | What it actually means |
|---|---|---|
| ICMP type 8/0 | "ping packets" | Echo Request (type 8) / Echo Reply (type 0) with id+seq, echoed verbatim |
| Time Exceeded | "traceroute's reply" | Type 11, code 0: a router decremented TTL to 0; its source reveals the hop |
| Dest Unreachable code 4 | "the MTU error" | "Frag Needed and DF Set" — carries next-hop MTU; the PMTUD signal |
| RFC 1071 checksum | "the ICMP checksum" | One's-complement 16-bit sum with field zeroed; must verify to 0 |
| Don't Fragment (DF) | "the bit that breaks PMTUD" | IP flag (RFC 791) telling routers not to fragment; set by PMTUD |
| Path MTU (PMTU) | "biggest packet that fits" | Smallest MTU along the path; found by DF probing + plateau convergence |
| Plateau table | "the MTU steps" | RFC 1191's MTU list; round a reported MTU up to the nearest plateau |
| Black-hole PMTUD | "big packets die" | A firewall drops type 3 code 4; large segments stall, small pass |
| MSS clamping | "the workaround" | Rewrite TCP MSS (RFC 6691) on SYN/SYN-ACK to PMTU−40 |
| TTL | "hop limit" | Time to Live / Hop Limit; reaching 0 yields type 11 code 0 |
