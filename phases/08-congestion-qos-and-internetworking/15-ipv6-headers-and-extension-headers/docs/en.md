# IPv6 Fixed Header and Extension Headers

> IPv6 collapses the IPv4 header from 13 fields and a variable 20–60 bytes into a **fixed 40-byte, 8-field** header: Version (4b = 6), Traffic Class (8b), Flow Label (20b), Payload Length (16b), Next Header (8b), Hop Limit (8b), Source Address (128b), Destination Address (128b). Everything IPv4 did in the main header — fragmentation, header checksum, options — is gone and reappears only as needed in a chain of **extension headers** identified by their Next Header values per RFC 8200: Hop-by-Hop Options (0), Routing (43), Fragment (44), Destination Options (60), AH (51, deprecated), ESP (50), Mobility (135). The recommended order is fixed header, Hop-by-Hop, Destination (intermediate), Routing, Fragment, Authentication, ESP, Destination (final), Upper-Layer. The network layer carries no checksum — L2 and L4 already cover integrity. Jumbograms (RFC 2675) extend the 65,535-byte ceiling via a Hop-by-Hop option. The Fragment header (8 bytes: Next Header, Reserved, Fragment Offset 13b, Res 2b, M 1b, Identification 32b) is the only place IPv6 carries fragmentation state, generated by the source. Path MTU Discovery (RFC 8200 §5) makes routers refuse to fragment: the path minimum is 1280 bytes.

**Type:** Build
**Languages:** Python, packet traces
**Prerequisites:** Binary, network byte order, the IPv4 header from §5.6.2.

## Learning Objectives

- **Pack the 40-byte fixed header.** Build the 8 fields per RFC 8200 §3 and unpack them back.
- **Walk a chain of extension headers.** From offset 40, follow `Next Header` pointers through Hop-by-Hop, Routing, Fragment, Destination Options, and into the upper-layer protocol; stop safely on a malformed chain.
- **Decode the three headers that matter most.** Parse Hop-by-Hop Options (TLV with the 2-bit "what if you don't understand me" action), Routing Type 0 (a list of intermediate IPv6 addresses plus a Segments Left counter), and Fragment (8 bytes with the 13-bit offset and M flag).
- **Build the upper-layer pseudo-header.** Construct the 40-byte pseudo-header from RFC 8200 §8.1 with the correct length, protocol, source, and destination.
- **Reason about the no-checksum and PMTUD choices.** Explain why IPv6 drops the IPv4 header checksum, what the 1280-byte path-MTU floor buys you, and what Jumbograms add.
- **Generate and inspect a real packet.** Run `main.py`, which builds a packet with Hop-by-Hop + Fragment + TCP, and confirm the bytes line up with the textbook diagrams.

## The Problem

You're staring at a `tcpdump -x` capture and the bytes after the link layer look different from what you remember. Only 40 bytes before the next protocol starts, no IHL, no Protocol, no Header Checksum, no Identification, no Flags, no Fragment Offset. The 16-byte addresses are unmistakable, but what follows is a string of `(next, len, ...)` triplets: the first is type `0` (Hop-by-Hop Options), the second is type `44` (Fragment), and the third says `6` — that's TCP, riding on top of two extension headers.

You need to parse the packet to find the TCP source port. The Fragment header's offset is split across three fields, the Hop-by-Hop TLV has a 2-bit action code in the high bits of the Type, and the addresses in a Type 0 Routing header can run in any order depending on `Segments Left`. You also need the upper-layer checksum, and IPv6 changed the pseudo-header — it now carries the *length of the upper-layer header* and the *upper-layer protocol number* (Next Header of the *final* header), not the IPv4-style "Protocol" inherited from the fixed header.

The lesson below shows how to encode, decode, and walk this structure in plain Python with `struct` and `bytes`.

## The Concept

### The fixed 40-byte header (RFC 8200 §3)

IPv6's main header is **fixed**: 40 bytes, 8 fields, always present. There is no IHL, no Option field, no Header Checksum, and no fragmentation state.

| Offset | Bits | Field | Meaning |
|-------:|-----:|-------|---------|
| 0      |    4 | Version | Always 6 for IPv6. |
| 0      |    8 | Traffic Class | Differentiated services; low 2 bits carry ECN. |
| 0      |   20 | Flow Label | Pseudo-connection identifier; random, hashed by routers. |
| 4      |   16 | Payload Length | Length of *everything after the 40-byte header*. 0 means a Jumbogram (RFC 2675) is in use. |
| 6      |    8 | Next Header | What follows: 0 = Hop-by-Hop, 6 = TCP, 17 = UDP, 41 = IPv6, 44 = Fragment, 50 = ESP, 51 = AH, 58 = ICMPv6, 60 = Destination Options, 135 = Mobility. |
| 7      |    8 | Hop Limit | Decremented at every hop; dropped at 0. |
| 8      |  128 | Source Address | 16 bytes, eight colon-separated 16-bit groups. |
| 24     |  128 | Destination Address | 16 bytes; if a Routing header is present, not necessarily final. |

Traffic Class carries DS codepoints (RFC 2474) and ECN (RFC 3168). The Flow Label is a 20-bit random tag combined with the address pair to identify a "flow" of packets that all want the same treatment. RFC 6437 specifies random generation; a source that gets back an ICMP error related to the flow should change the label and retry. The fixed length lets a forwarding-only router read the destination address at a fixed offset and ship the packet on with a single 16-byte read.

### Hop-by-Hop, Routing, Fragment, Destination Options, AH, ESP

The Next Header field is the keystone. A fixed header with `Next Header = 0` means "the next 8 bytes start a Hop-by-Hop header that itself has a Next Header pointing further along." Walking the chain is mechanical: read one byte (the type), use it to find the format and length of *this* header, advance, repeat.

RFC 8200 §4 recommends this order when multiple extension headers are present:

1. IPv6 fixed header → Hop-by-Hop Options → Destination Options (intermediate) → Routing → Fragment → AH → ESP → Destination Options (final) → Upper-Layer

**Hop-by-Hop (Next Header 0)** is "for information that all routers along the path must examine." Format: an 8-bit Next Header, an 8-bit Header Extension Length (in 8-byte words *excluding* the first 8 bytes), then zero or more options. Each option is a TLV — **Type (1 byte)**, **Length (1 byte)**, **Value (0 to 255 bytes)** — except for the 1-byte Pad options. The Type byte's high 2 bits encode the action a router must take when it doesn't recognize the option: `00` skip, `01` discard, `10` discard+ICMPv6 Parameter Problem, `11` discard+ICMPv6-only-unicast. The textbook's only defined use was the **Jumbo Payload** option (type 194, action `11`); when present, Payload Length is forced to 0 and the option's 4-byte value gives the real length, up to 4,294,967,295 bytes. The **Router Alert** option (type 148, action `00`) is used by RSVP and IGMPv3.

**Routing (Next Header 43)** is the loose-source-routing revival. The textbook covers **Type 0**: a 4-byte fixed prefix (Next Header, Header Extension Length, Routing Type, Segments Left), a 4-byte reserved word, then a list of IPv6 addresses. `Segments Left` starts at the number of addresses and counts down: when a router decrements it to zero, the destination has been reached. Type 0 was deprecated by RFC 5095 because of amplification attacks. RFC 8200 standardizes only **Type 3** (Source Routing) and **Type 4** (Segment Routing, RFC 8407).

**Fragment (Next Header 44)** is the only place IPv6 carries fragmentation state, and it is **8 bytes fixed**:

| Offset | Bits | Field |
|-------:|-----:|-------|
| 0 | 8 | Next Header |
| 1 | 8 | Reserved |
| 2 | 13 | Fragment Offset (in 8-byte units) |
| 3.625 | 2 | Res |
| 3.875 | 1 | M (more fragments) |
| 4 | 32 | Identification |

The Identification field is a 32-bit label that ties all the fragments of one original packet together. **Only the source can fragment** IPv6 packets. If a router receives a packet that is too big for the outgoing link, it drops the packet and sends an ICMPv6 "Packet Too Big" message back to the source.

**Destination Options (Next Header 60)** is structurally identical to Hop-by-Hop, but appears in the chain only for nodes listed in any preceding Routing header and for the final destination.

**Authentication Header (Next Header 51)** and **Encapsulating Security Payload (Next Header 50)** are the IPsec mechanisms. AH authenticates the entire packet (except for fields that must change in transit, like Hop Limit) and is **deprecated by RFC 8200** because ESP with authentication provides the same integrity guarantee plus confidentiality. When a packet uses ESP, the upper-layer checksum's pseudo-header still references the **outer** IPv6 source and destination.

### No header checksum

RFC 8200 §3 makes a deliberate omission: no Header Checksum field. The textbook's design discussion is candid: "any application that really cares about data integrity has to have a transport layer checksum anyway, so having another one in IP is overkill." Dropping the field means a router can decrement the Hop Limit and forward without recomputing a 16-bit sum over the entire header.

IPv6 still requires the upper-layer checksum — TCP, UDP, ICMPv6 — be computed over a **pseudo-header** that includes the IPv6 source and destination. The pseudo-header changed in two ways: the length field is the *length of the upper-layer data* (not the total IP length), and the protocol value is the **final** Next Header value, not whatever the fixed header carried.

### Path MTU Discovery, Jumbograms, and the 1280-byte floor

The IPv6 minimum link MTU is **1280 bytes**, up from IPv4's 576. A link that can't forward 1280 bytes must do something at a lower layer (fragment the link frame) or use tunneling; it cannot be a normal IPv6 link.

PMTUD: the source sends the largest packet it can, with the implicit "don't fragment" rule of IPv6 baked in. A router that can't forward it drops the packet and returns ICMPv6 Packet Too Big with the link's MTU. The source tries again with that MTU, and repeats until the packet gets through. If a link is below 1280, IPv6 doesn't run on it as a native link layer (RFC 8200 §5).

For payloads above 65,535 bytes, the Jumbogram Hop-by-Hop option (RFC 2675) raises the ceiling to 4 GB. When this option is present, Payload Length is 0 and the option carries the real length.

## Build It

`code/main.py` implements an IPv6 fixed-header encoder/decoder, parsers for the three extension headers the lesson emphasizes (Hop-by-Hop, Routing Type 0, Fragment), a chain walker, and the upper-layer pseudo-header. The `__main__` block assembles a packet with Hop-by-Hop + Fragment + TCP and walks the chain.

```python
from main import (
    encode_fixed_header, encode_hop_by_hop, encode_fragment,
    decode_fixed_header, decode_hop_by_hop, decode_routing_type0,
    decode_fragment, walk_chain, build_pseudo_header,
)

# Build a packet by hand.
src = b"\x20\x01\x0d\xb8" + b"\x00" * 12  # 2001:db8::
dst = b"\x20\x01\x0d\xb8" + b"\x00" * 11 + b"\x01"
fixed = encode_fixed_header(
    version=6, traffic_class=0, flow_label=0,
    payload_length=0,        # filled in later
    next_header=0,           # Hop-by-Hop
    hop_limit=64, src=src, dst=dst,
)
hbh = encode_hop_by_hop(next_header=44, options=[])
frag = encode_fragment(next_header=6, offset=0, m_flag=True, identification=0xDEADBEEF)
tcp = b"\x00\x50\x00\x51" + b"\x00" * 12  # src=80, dst=81, no payload

packet = fixed + hbh + frag + tcp
length = len(packet) - 40
packet = fixed[:4] + length.to_bytes(2, "big") + fixed[6:]

for header in walk_chain(packet):
    print(header)
ps = build_pseudo_header(src, dst, len(tcp), next_header=6)
```

`walk_chain` returns typed records: `("fixed", ...)`, `("hop_by_hop", ...)`, `("routing", ...)`, `("fragment", ...)`, and `("upper", protocol_number)` for non-extension-header protocols. It stops safely on a malformed chain.

Run with `python3 code/main.py` to see the chain walk printed for a Hop-by-Hop + Fragment + TCP packet. Modify `__main__` to build other chains (Routing + Destination Options + UDP, say) and confirm the output matches the diagrams in the textbook.

## Ship It

- **Strict width checks.** Every decoder asserts that its input is the right length. A 39-byte fixed header is a programming error, not a runtime condition; `decode_fixed_header` raises `ValueError` immediately.
- **No third-party imports.** The whole module uses only `struct` and `bytes` operations.
- **Padded options.** `encode_hop_by_hop` always pads to an 8-byte boundary. Per RFC 8200, extension headers should be an integer multiple of 8 bytes long.
- **Type 0 is intentionally limited.** The encoder and decoder cover Type 0 because the textbook diagrams it. The deprecated `Routing Type 0` warning is noted in the docstring; don't generate them in production. The decoder still parses them so old captures keep working.
- **Pseudo-header is built, not checksummed.** `build_pseudo_header` produces the bytes that go *into* the checksum. The actual checksum is the caller's job.
- **Constants live in one place.** Next Header numbers (0, 6, 17, 41, 44, 50, 51, 58, 60, 135) are defined as module-level constants so the names in the chain walker are self-explanatory and the values don't drift across the file.

## Key Terms

| Term | What it actually means |
|------|-----------------------|
| Fixed header | 8 fields: Version, Traffic Class, Flow Label, Payload Length, Next Header, Hop Limit, Source Address, Destination Address. No checksum, no IHL, no fragmentation. |
| Extension header | Optional headers identified by Next Header value: Hop-by-Hop (0), Routing (43), Fragment (44), Destination Options (60), AH (51), ESP (50), Mobility (135). |
| Next Header | 8-bit field in every header. In the fixed header it points to the first extension header (or the upper layer if none). |
| Hop-by-Hop Options | TLV-encoded options in extension-header type 0. The Type byte's high 2 bits tell routers what to do if they don't understand: skip, discard, discard+ICMP, or discard+ICMP-only-unicast. |
| Routing header | Extension-header type 43. RFC 8200 standardizes Type 3 and Type 4; Type 0 (the textbook example) is deprecated by RFC 5095 because of amplification attacks. |
| Fragment header | 8-byte extension-header type 44. Carries Next Header, Fragment Offset (13b, in 8-byte units), M flag, and Identification (32b). |
| Flow Label | 20-bit tag in the fixed header, combined with src/dst addresses, identifies a flow. RFC 6437 specifies random generation. |
| Payload Length | 16-bit count of bytes after the 40-byte fixed header. 0 means a Jumbogram is in use. |
| Hop Limit | 8-bit counter, decremented at every hop, packet dropped at zero. 255 hops is plenty. |
| Jumbogram | A packet with a Hop-by-Hop Jumbo Payload option (type 194). Payload Length is 0 and the option's 4-byte value gives the real length, up to 4 GB. RFC 2675. |
| Path MTU Discovery | Source sends a packet that's too large; a router drops it and returns ICMPv6 Packet Too Big with the link's MTU; source retries. Minimum link MTU is 1280. |
| Pseudo-header | 40-byte structure of src/dst addresses, upper-layer length, zero pad, and protocol number. Built from the *final* Next Header value. |
| Upper layer | The protocol identified by the *last* Next Header in the chain. |
| ICMPv6 | Protocol 58. Replaces ICMP, ARP, and IGMP for IPv6. |

## Further Reading

- [RFC 8200] Deering, S. and Hinden, R. "Internet Protocol, Version 6 (IPv6) Specification." The current normative reference. §3 (fixed header), §4 (extension headers), §4.5 (order), §5 (PMTUD), §8.1 (pseudo-header). Replaces RFC 2460.
- [RFC 4291] Hinden, R. and Deering, S. "IP Version 6 Addressing Architecture." 128-bit unicast/multicast, `::` compression, IPv4-mapped, `fe80::/10`, `fc00::/7`, global unicast prefix.
- [RFC 2675] Borman, D., Deering, S., and Hinden, R. "IPv6 Jumbograms." The Hop-by-Hop Jumbo Payload option (type 194) and the 4 GB payload ceiling.
- [RFC 6437] Amante, S., Carpenter, B., and Jiang, S. "IPv6 Flow Label Specification." Tightens the rules for Flow Label generation (random, hashable) and re-use.
