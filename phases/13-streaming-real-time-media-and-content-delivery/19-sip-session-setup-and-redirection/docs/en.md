# SIP Session Setup and Redirection

> The Session Initiation Protocol is the signaling layer of VoIP: it locates the called party, rings their device, negotiates the media parameters, and tears the call down. It is text-based, modeled on HTTP, and built for redirection so a call can follow a user anywhere.

## The Problem

A phone call has two halves: the signaling (ring, answer, hang up, redirect) and the media (the voice samples carried by RTP). SIP handles the signaling. The challenge is that the called party may be anywhere: at a desk phone, on a softphone, roaming to a new IP, or behind a NAT. SIP must locate the user, ring one or more devices in parallel, negotiate a common codec and port pair, and then get out of the way so RTP flows directly between the two endpoints. It must do this with text messages that look like HTTP, over UDP or TCP, while handling redirects, voicemail fallback, and call transfers. Understanding SIP means understanding a handful of message types, a set of response codes borrowed from HTTP, and the offer/answer model for describing media in SDP.

## The Concept

### SIP architecture roles

A SIP network has four logical roles, often combined in one box:

- **User Agent (UA)**: an endpoint, like a softphone or desk phone. Acts as UAC (client, sends requests) or UAS (server, sends responses) depending on direction.
- **Registrar**: receives REGISTER messages binding a user's SIP URI to a contact address. Stores the binding in a location service.
- **Proxy server**: forwards requests toward the callee, possibly forking to multiple contacts. A stateful proxy remembers transaction state; a stateless proxy does not.
- **Redirect server**: responds to an INVITE with a 3xx redirect pointing the caller at a new location, instead of forwarding the request itself.

### Message types

SIP requests (methods) and their purpose:

```text
INVITE     initiate a call (ring the far end)
ACK        confirm that the caller received the 200 OK (3-way handshake)
BYE        terminate the call
REGISTER   bind a SIP URI to a contact address
CANCEL     stop a pending INVITE before it is answered
OPTIONS    query capabilities (codecs, methods supported)
REFER      transfer the call to another party
INFO       mid-call signaling (DTMF, events)
```

### Response codes

SIP response codes reuse the HTTP first-digit convention:

```text
1xx Provisional   100 Trying, 180 Ringing, 183 Session Progress
2xx Success       200 OK
3xx Redirection   302 Moved Temporarily
4xx Client error  404 Not Found, 486 Busy Here, 487 Request Terminated
5xx Server error  503 Service Unavailable
6xx Global error  603 Decline
```

### The basic call flow

```text
Caller (UAC)          Proxy          Callee (UAS)
   |                    |                |
   |--- INVITE -------->|                |
   |<-- 100 Trying -----|                |
   |                    |--- INVITE ----->|
   |                    |<-- 180 Ringing--|
   |<-- 180 Ringing ----|                |
   |                    |<-- 200 OK ------|
   |<-- 200 OK ---------|                |
   |--- ACK ----------->|--- ACK -------->|
   |                    |                |
   |==== RTP media flows directly caller <-> callee ====|
   |                    |                |
   |--- BYE ----------------------------------------->|
   |<-- 200 OK ---------------------------------------|
```

The INVITE/200/ACK is a three-way handshake (unlike HTTP's two-way), because the 200 OK is generated by the callee, not a network relay, and the caller must confirm receipt so the callee stops ringing.

### Registration

Before a call, the callee registers their current location:

```text
Callee -> Registrar:  REGISTER sip:alice@example.com
                         Contact: <sip:alice@192.0.2.10:5060>
Registrar -> Callee:   200 OK
```

Now the proxy can look up `alice@example.com` in the location service and find `sip:alice@192.0.2.10:5060`. Registration expires (default 3600 s); the UA must re-register before the binding lapses or the proxy will return 404.

### Redirection

A redirect server does not forward the INVITE. It returns a 302 with a Contact header pointing at a new destination:

```text
Caller -> Redirect:  INVITE sip:bob@example.com
Redirect -> Caller:   302 Moved Temporarily
                        Contact: <sip:bob@mobile.example.net>
Caller -> Mobile:    INVITE sip:bob@mobile.example.net
```

The caller then issues a new INVITE to the redirected contact. This keeps the redirect server stateless and lets the caller decide whether to follow the redirect.

### Forking

A proxy may fork an INVITE to multiple registered contacts in parallel (simultaneous ring on desk phone and mobile). The first contact to answer with 200 OK wins; the others get a CANCEL. This is how "ring all my devices" works.

### SDP: the offer/answer model

The INVITE body carries a Session Description Protocol (SDP) offer describing the caller's media: codecs, ports, RTP payload types. The 200 OK carries the answer, selecting from the offered codecs. Media then flows on the agreed ports.

```text
v=0
o=alice 12345 1 IN IP4 192.0.2.10
s=-
c=IN IP4 192.0.2.10
t=0 0
m=audio 5004 RTP/AVP 0 8       ; payload types 0 (PCMU/G.711u) and 8 (PCMA/G.711a)
a=rtpmap:0 PCMU/8000
a=rtpmap:8 PCMA/8000
```

### Failure modes

- **No response / timeout**: INVITE sent, no 100 Trying or final response. Caller retransmits (UDP) then gives up with a 408.
- **Loop detected**: a proxy adds a Via header; if it sees its own address again, it returns 482 Loop Detected.
- **Codec mismatch**: SDP offer has no codec in common with the answer; call fails with 488 Not Acceptable Here.
- **NAT binding expiry**: SIP over UDP keeps a NAT pinhole open only while packets flow; if registration lapses, inbound INVITEs fail. Keep-alives refresh the binding.
- **One-way audio**: signaling succeeded but one side's RTP port is unreachable (NAT, firewall); the call connects but only one direction has audio.

## Key Terms

| Term | What people say | What it actually means |
|------|----------------|------------------------|
| SIP | "VoIP signaling" | Session Initiation Protocol: text-based signaling to locate, ring, answer, and tear down media sessions |
| User Agent | "A phone" | A SIP endpoint that acts as client (UAC) or server (UAS) per transaction |
| Registrar | "A directory" | A server that accepts REGISTER bindings mapping a SIP URI to a current contact address |
| Proxy | "A relay" | A server that forwards requests toward the callee, possibly forking to multiple contacts |
| Redirect server | "A pointer" | A server that returns a 3xx with a new Contact instead of forwarding, leaving the caller to re-INVITE |
| INVITE | "A ring" | The SIP request that initiates a session; carries the SDP offer in its body |
| Three-way handshake | "INVITE/200/ACK" | The caller's ACK confirms receipt of 200 OK so the callee stops ringing; unlike HTTP's two-way |
| SDP | "The media spec" | Session Description Protocol: the body that lists codecs, ports, and payload types via offer/answer |
| Forking | "Ring all" | A proxy sending one INVITE to multiple contacts in parallel; first 200 OK wins, others are cancelled |
| Via header | "Loop guard" | A header each proxy adds; seeing its own address again means a loop (482 Loop Detected) |
