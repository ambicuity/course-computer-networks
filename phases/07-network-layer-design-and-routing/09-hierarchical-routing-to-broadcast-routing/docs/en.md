# Hierarchical Routing to Broadcast Routing

> Two scaling problems collide in the network layer. First, a flat routing table grows linearly with the network: 720 routers means 720 entries per router, consuming memory, CPU scan time, and link bandwidth for status updates. Hierarchical routing fixes this by splitting routers into regions, clusters, and zones — a router keeps full detail for its own region and a single condensed entry per remote region, collapsing 1A's table from 17 to 7 entries in Tanenbaum's Fig. 5-14, at the cost of slightly longer paths (1A→5C goes via region 3, not the optimal region 2). Kamoun and Kleinrock (1979) proved the optimal hierarchy depth is ln N levels with e·ln N entries per router. Second, broadcasting a packet to every host is needed for weather feeds, OSPF link-state flooding, and ARP-style discovery. The naive approach sends N separately-addressed copies; multidestination routing packs a bitmap; flooding with per-source sequence numbers is robust but duplicative; reverse path forwarding (Dalal & Metcalfe, 1978) uses the unicast table — accept a copy only if it arrived on the link you'd use to reach the source, else discard as a duplicate — sending each link once per direction with no sequence-number state. Spanning-tree broadcast is optimal at exactly N−1 packets but needs every router to know a shared tree. This lesson builds a Python tool that computes hierarchical tables, simulates RPF, and counts packets so you can see the trade-offs as numbers.

**Type:** Build
**Languages:** Python, routing traces
**Prerequisites:** Phase 7 lessons on shortest-path routing (Dijkstra, sink trees), distance-vector and link-state basics

## The Problem

You operate a backbone that has grown from 200 to 1,800 routers over three years. Two pages fire in the same week. The first: route processors are hitting 90% memory and convergence after a link flap now takes 40+ seconds because every router carries a full 1,800-entry table and floods the whole thing on change. The second: a new telemetry service wants to push a config-snapshot to *all* routers every 60 seconds, and the ops team naively unicasts 1,800 copies from one box, saturating its uplink and the links nearest the source.

Both are routing-scale problems with classic answers. The first is solved by introducing a routing *hierarchy* so each router stores `O(e·ln N)` entries instead of `O(N)`. The second is a *broadcast routing* problem: you do not want 1,800 unicast copies, and you do not want naive flooding's duplicate storm — you want reverse path forwarding or a spanning tree. Getting these wrong shows up as table bloat, slow convergence, broadcast storms, and saturated near-source links. This lesson makes the trade-offs concrete with code you can run.

## The Concept

### Why flat routing tables stop scaling

A router's forwarding table holds one entry per reachable destination. In a flat design with `N` routers, that is `N` entries everywhere. Three costs grow with the table:

- **Memory:** every entry consumes TCAM/RAM in the route processor.
- **CPU:** longest-prefix-match scans and route recomputation are proportional to table size.
- **Bandwidth:** link-state or distance-vector updates carry table state; bigger tables mean bigger, more frequent advertisements.

The fix mirrors the telephone network: route *hierarchically*. Divide routers into **regions**. A router knows full detail for destinations inside its own region but treats every other region as a single condensed destination reachable through one egress line. For huge networks you nest further — regions into **clusters**, clusters into **zones** — until you run out of names for aggregations.

### Hierarchical table reduction — the 1A worked example

Tanenbaum's Fig. 5-14 uses five regions. Router 1A's *full* table has 17 entries (one per other router). The *hierarchical* table keeps the local region's routers explicit and collapses each remote region to one entry:

| Dest. | Line | Hops | Note |
|-------|------|------|------|
| 1A    | —    | —    | self |
| 1B    | 1B   | 1    | local, explicit |
| 1C    | 1C   | 1    | local, explicit |
| Region 2 | 1B | 2 | aggregated — all of 2A/2B/2C/2D |
| Region 3 | 1C | 2 | aggregated |
| Region 4 | 1C | 3 | aggregated |
| Region 5 | 1C | 4 | aggregated |

That is **7 entries instead of 17**. The penalty is path length: the genuinely shortest route from 1A to 5C runs through region 2, but the aggregated entry sends *all* region-5 traffic via region 3, because region 3 is the better egress for most region-5 destinations. You accept a worse path for one destination to shrink the table.

### How many levels? The ln N rule

Consider a 720-router network. `code/main.py` reproduces this table:

| Design | Local | Same-cluster regions | Distant clusters | Total entries |
|--------|-------|----------------------|------------------|---------------|
| Flat (no hierarchy) | 720 | — | — | **720** |
| Two-level: 24 regions × 30 routers | 30 | — | 23 remote regions | **53** |
| Three-level: 8 clusters × 9 regions × 10 routers | 10 | 8 | 7 | **25** |

Kamoun and Kleinrock (1979) proved the optimum: for an `N`-router network use about `ln N` levels, giving roughly `e·ln N` entries per router. For N=720, `ln 720 ≈ 6.6` levels and `e·ln 720 ≈ 18` entries — the asymptotic floor the discrete designs above are approaching. They also showed the resulting mean-path-length increase is small enough to be acceptable in practice.

### Broadcasting: four mechanisms, four packet counts

Broadcasting means delivering one packet to *every* host — weather feeds, stock tickers, OSPF link-state flooding, ARP discovery. Four classic methods, ordered by efficiency:

| Method | Source needs | Router state | Packets (Fig. 5-15) | Failure mode |
|--------|--------------|--------------|---------------------|--------------|
| Separate addressing | full destination list | none | N unicast copies | wasteful, slow, source must enumerate all |
| Multidestination routing | full list / bitmap in header | none | < N (shared legs ride free) | header grows with group; per-hop split work |
| Flooding (seq# per source) | nothing | seq-number cache per source | many duplicates | duplicate storm without hop-count/seq guard |
| Reverse path forwarding | unicast table only | next-hop-to-source only | 24 here | extra copies vs. tree |
| Spanning-tree broadcast | shared spanning tree | tree-membership per line | 14 (= N−1) | needs shared tree (hard in distance-vector) |

### Reverse path forwarding — the accept/discard rule

RPF (Dalal & Metcalfe, 1978) is elegant: when a broadcast packet arrives at a router, check the link it came in on. **If that link is the one this router would normally use to send unicast traffic *toward the source*, accept it and forward copies onto every other link. Otherwise, discard it as a likely duplicate.** The intuition: a copy arriving on the reverse of your best path to the source almost certainly travelled the best path *from* the source, so it is the first, canonical copy.

What makes RPF practical is the *state it does not need*. It reuses the existing unicast forwarding table — no per-source sequence-number cache (unlike flooding), no destination list in the header (unlike multidestination), no shared spanning tree (unlike tree broadcast). Each link carries the broadcast exactly once in each direction.

On Fig. 5-15's network, broadcasting from router I: hop 1 reaches F, H, J, N (all on preferred links — circled in the sink tree of `assets/hierarchical-routing-to-broadcast-routing.svg`). By hop 5 RPF has sent **24 packets**; the same broadcast over the exact sink tree needs only **14**. RPF trades roughly 10 extra packets for zero extra router state.

### Spanning-tree broadcast — the optimal floor

A **spanning tree** is a loop-free subgraph touching every router; a sink tree is one example. If each router knows which of its lines belong to the tree, it copies an incoming broadcast onto all tree lines *except* the arrival line. This generates the absolute minimum — `N−1` packets (14 in Fig. 5-15) — because every router receives exactly one copy. The catch is the prerequisite: every router must agree on a tree. Link-state routing makes this easy (all routers hold the full topology and can compute the same tree deterministically); distance-vector routing does not, which is why RPF is the pragmatic default when no shared tree exists.

`code/main.py` simulates RPF and spanning-tree broadcast on a small topology and prints the per-hop packet counts so you can watch 24 vs N−1 emerge.

## Key Terms

| Term | What people say | What it actually means |
|------|-----------------|------------------------|
| Region | "a subnet" | A set of routers whose internal topology is hidden from outside routers; the unit of hierarchical aggregation |
| Hierarchical routing | "summarization" | Storing full detail for your own region and one condensed entry per remote region, giving `O(e·ln N)` table size |
| Path-length penalty | "hierarchy is lossy" | Aggregation forces all traffic to a region through one egress, suboptimal for some individual destinations |
| Reverse path forwarding | "RPF" | Accept a broadcast copy only if it arrived on the link you'd use to reach its source; discard otherwise |
| Sink tree | "shortest-path tree" | The union of shortest paths from all routers to one destination; loop-free, used as a spanning tree |
| Spanning tree | "a loop-free overlay" | A subgraph touching every router with no loops; broadcast over it costs exactly N−1 packets |
| Multidestination routing | "a packet with many addresses" | A packet carrying a destination bitmap/list that routers split per output line |
| Flooding | "send it everywhere" | Forward each packet on every link except arrival; needs a per-source sequence-number cache to terminate |
